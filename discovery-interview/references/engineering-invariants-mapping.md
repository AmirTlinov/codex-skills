# Mapping: управленческая цель → инженерные инварианты

| Блок вопроса (plain) | Инвариант (что обязательно) | Как проверить |
| --- | --- | --- |
| Цель/результат | Есть явная цель и ожидаемый эффект | Проверяется формулировкой в одной строке в <proposed_plan> |
| Критерий успеха | Есть измеримый критерий успеха | Есть метрика, дедлайн и владелец проверки |
| Участники | Есть роли и точки принятия решений | Указаны: кто решает/кто подтверждает |
| Ограничения | Ограничения зафиксированы как hard constraints | Любая реализация обязана их упомянуть и не нарушать |
| Риски | Приоритетные риски и смягчающие действия | Есть mitigation и ответственный |
| Приоритеты | Ясная очередность выполнения | Slice-структура отражает порядок и зависимости |

## Правило инвариантов для слайсов

- Каждый слайс должен менять максимум 1–2 инварианта.
- Слайс считается готовым только при закрытии всех инвариантов блока.
- Следующий слайс не стартует до подтверждения предыдущего.

## Пример mapping

- Бизнес-вопрос: «Главное — быстро запустить в пилоте»
- Инвариант: `MVP до 2 недель, без влияния на существующий процесс.`
- Проверка: в слайсе 1 есть ограничение на изменение существующих модулей и оценка «когда запустим».
