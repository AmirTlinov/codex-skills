# Форматы плана (обязательный выбор)

## 1) GitHub Issues

- Использовать когда нужна прозрачная постановка для трекинга.
- Формируем 3 блока:
  - **Epic/Parent:** одна строка с целью и ценностью.
  - **Child issues:** последовательность зависимых подзадач.
  - **Acceptance:** что проверяем и кто принимает решение.
- Результат в `<proposed_plan>` — короткий summary, что будет сгенерировано как issue-структура.

## 2) PLAN+SLICE (рекомендуемый)

- Формируем один короткий `PLAN.md` (план-лист) и 1+ `SLICEn.md`.
- Каждый слайс должен быть итеративным:
  - `Slice n` решает конкретный набор инвариантов,
  - `Slice n+1` строится только после подтверждения `Slice n`.
- Результат в `<proposed_plan>` — индекс артефактов и ожидаемые выходы по каждому слайсу.

## 3) full <proposed_plan>

- Подробный единый план в сообщении:
  - Контекст
  - Список решений
  - Риски и зависимости
  - Проверки/критерии приемки
  - Дальнейшие шаги
- Результат в `<proposed_plan>` — полный план + `self-critique`.

## Единое правило

- Для всех форматов обязательна `self-critique`.
- Формат и стиль выдачи всегда управленческий, без перегруза инженерными деталями.
