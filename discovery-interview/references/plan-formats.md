# Форматы плана (обязательный выбор)

## 1) PLAN+SLICE (рекомендуемый)

- Использовать, когда важно **не расписывать “простыню” в чате**, а держать план в файлах.
- Папка плана (в текущем проекте): `docs/plans/{plan_slug}/`
  - `PLAN.md` — короткий индекс: цель, рамки, метрики успеха, список слайсов
  - `SLICE1.md` — первая итерация (делаем до конца)
  - `SLICE2.md+` — следующие итерации (расписываем после подтверждения SLICE1)
- Слайсы должны быть итеративными (dependency-aware), а не «рандомным списком»:
  - `Slice n` закрывает 1–2 ключевых инварианта
  - `Slice n+1` уточняется только после подтверждения `Slice n`
- Результат в `<proposed_plan>` — **короткий** summary + индекс файлов + self-critique + flagship-suggestions.

## 2) GitHub Issues

- Использовать, когда нужна прозрачная постановка задач для трекинга и делегирования.
- Формируем:
  - **Parent issue:** цель + ценность + критерий успеха (1 экран текста максимум)
  - **Child issues:** итеративная цепочка (Child 2 зависит от Child 1)
  - **Acceptance:** как проверяем и кто подтверждает
- Результат в `<proposed_plan>` — короткий summary + список issues + self-critique + flagship-suggestions.

## 3) full <proposed_plan>

- Подробный единый план в сообщении (используй только если пользователь прямо просит):
  - Контекст
  - Список решений
  - Риски и зависимости
  - Проверки/критерии приемки
  - Дальнейшие шаги
- Результат в `<proposed_plan>` — полный план + `self-critique` + `flagship-suggestions`.

## Единое правило

- Для всех форматов обязательны `self-critique` и `flagship-suggestions`.
- Формат и стиль выдачи всегда управленческий, без перегруза инженерными деталями.
