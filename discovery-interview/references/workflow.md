# Workflow: как вести discovery-интервью

## Карта процесса

1. **Стартовый контур**
   - Подтверждаем цель и что работаем именно в формате discovery.
   - Проверяем Plan Mode.
2. **Интервью-модуль**
   - 2–3 коротких блока вопросов через `request_user_input`.
   - После каждого блока: мини-сводка + decision log (см. `references/decision-log.md`).
   - Не задаём повторные вопросы, если ответ уже зафиксирован (повторяем только при противоречии).
3. **Формат-переход**
   - Обязательный выбор: `PLAN+SLICE` (recommended/default), `GitHub Issues`, `full <proposed_plan>`.
4. **Итоговое преобразование**
   - Формируем короткий `Intent Profile` (1–2 абзаца).
   - Строим карту «обязательных условий» (инвариантов) (см. `references/engineering-invariants-mapping.md`).
   - Возвращаем `<proposed_plan>` + `self-critique` + `flagship-suggestions`.

## Контрольный список на каждом шаге

- Собраны ли цели и ценность?
- Сформулированы ли критерии «что обязательно будет сделано»?
- Есть ли приоритеты и ограничения?
- Указаны ли риски и что критично для дедлайна?
- Выбран ли формат плана через `request_user_input`?

## Что делать при недостающих фактах

- Не продолжать с предположениями.
- Снова вызвать `request_user_input` с одним уточняющим блоком.
- Дождаться ответа и только потом продолжать.

## Финальная проверка перед выдачей

- Есть выбранный формат.
- Есть map из управленческого языка в «обязательные условия» (инварианты).
- Есть план в `<proposed_plan>` с короткой самокритикой и `flagship-suggestions` (даже для форматов 1/2).
