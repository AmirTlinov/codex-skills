# Discovery Interview

## Быстрое назначение

Этот скилл помогает менеджеру объяснить задачу простым языком, а затем превращает ответы в понятный технический план. Работает **только в Plan Mode**.

## Обязательные правила запуска

1. **Plan Mode gate**

- Перед любым действием скилл проверяет, что текущий режим — **Plan Mode** (режим планирования).
- Плановый режим определяется **не** по доступности `request_user_input`, а по системному уведомлению о режиме (например, «You are now in Plan mode» / «Collaboration Mode: Plan»).
- Если текущий режим **не** Plan Mode — fail-closed: ответь одной фразой и остановись: «Включи, пожалуйста, Plan Mode и перезапусти шаг — этот скилл работает только в Plan Mode».

2. **Все вопросы только через `request_user_input`**

- 1–3 вопроса за один вызов.
- 2–3 варианта ответа.
- Без ручного `Other`.

3. **Минимум инженерного жаргона**

- Формулировки ориентированы на управленцев: бизнес-результат, риски, приоритеты, сроки.
- Технический язык используется только в разделе mapping в инварианты (чётко и коротко).

## Как использовать скилл (план выполнения)

### Шаг 1 — входная валидация и рамки

- Прочитай цель задачи одним предложением и зафиксируй, что это **discovery-сессия**.
- Убедись, что задача в Plan Mode и у пользователя понятен формат дальнейшей работы.

### Шаг 2 — проведение интервью

- Используй вопросы из `references/question-bank.md`.
- Собери ответы в `Intent Profile` (1–2 абзаца).
- Если после базовых блоков осталось непонятно — задай дополнительные уточняющие вопросы через `request_user_input` (до 20 вопросов суммарно), пока намерение не станет достаточно ясным для исполнимого плана.
- Отдельно спроси и зафиксируй «что НЕ делаем сейчас» (это станет явным OUT в плане/задачах).
- После каждого блока:
  - зафиксируй decision log (см. `references/decision-log.md`)
  - свяжи ответы с «обязательными условиями» (инвариантами) через `references/engineering-invariants-mapping.md`.

### Шаг 3 — обязательный выбор формата

- Обязательно спроси формат плана через `request_user_input` (рекомендуемый вариант: **PLAN+SLICE (Recommended)**):
  1. `PLAN+SLICE`
  2. `GitHub Issues`
  3. `full <proposed_plan>`

### Шаг 4 — финальная выдача

- **Всегда** сформируй блок `<proposed_plan>`.
- Для форматов `GitHub Issues` и `PLAN+SLICE` вывод — короткий summary + карта артефактов + `self-critique` + `flagship-suggestions`.
- Для `full <proposed_plan>` вывод — детальный план + `self-critique` + `flagship-suggestions`.
- Для каркаса структуры используй шаблоны из `references/templates/`.

## Дополнительные ориентиры

- Слайды формата `PLAN+SLICE` строятся итеративно (каждый слайс опирается на предыдущий, а не случайный список).
- Если критические вводные не получены, вернись к нужному вопросу через `request_user_input`, а не придумывай допущения.
- Окончательно зафиксируй: цель, рамки, приоритеты, метрики успеха, зависимости и ограничения.
